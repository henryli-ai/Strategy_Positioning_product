Battle Strategy｜我們打哪裡、槓桿哪裡
Where We Fight

雲端模型的智力，我們用 Model Routing 槓桿；
地端的機敏，我們用主權與治理獨佔。

不是每個對手都要正面打。雲端通用助理（ChatGPT、Claude、Gemini）和 SaaS 巨頭（Microsoft、Salesforce）的模型與生態，我們打不贏也不必打——它們的模型強度反而是我們透過 routing 接進來的資產。真正要贏的戰場是它們碰不到的位置：資料不出企業的地端機敏場景，以及把路由、硬體、治理綁成一套的可採購產品。下方競品由「最該贏」到「不必硬碰」排序。

🎯 主戰場：台灣地端 + 治理 ⚔️ 正面交鋒：雲地混合平台 🔧 借力：Routing 層 🤝 槓桿不對決：雲端通用助理
Market Structure｜全球市場五層結構

以買方實際採購的「產品包裝」為單位盤點代表性產品。結論：成熟的戰場不在「再做一個 Agent Builder」，而在「治理 + 部署彈性 + Model Routing + 成本控制」的控制平面層。層級由下（個人）至上（全公司級系統），但競爭優先序與層級高低無關——見下方排序。

LAYER 0

雲端通用助理
（模型即服務）

代表玩家ChatGPT Enterprise、Claude Enterprise、Gemini；OpenAI Frontier
對 OrientAI 的意義模型最強、生態最大，但資料進雲、無地端選項。不對決——用 routing 把它們的智力接進來，當客戶問「直接買 ChatGPT 不就好」時，用地端機敏與資料主權切分訴求。
LAYER 1

System-of-Record 型
SaaS Agent 套件

代表玩家Microsoft、Salesforce、ServiceNow、Oracle、SAP、Workday

LAYER 2

雲地混合控制平面
與 Agent 編排平台

LAYER 3

Routing-first

Self-hosted Agent
個人／小隊 Runtime

Sales Narrative｜三句話講完產品主張
第一句

「你不用在公有雲和地端之間二選一。」

企業不是不想上雲，是不想把所有任務都上雲。例行問答、PII、延遲敏感留地端；長文推理、外部搜索、特殊模型再上雲。

第二句

「你不是在選模型，而是在選一個會自動替你選模型的控制平面。」

Portkey、LiteLLM、Not Diamond 已驗證市場願為路由付費——但它們不從企業本地硬體分層出發。路由是硬體的附加價值，不是獨立 API 小工具。

第三句

「你買的不只是 Agent，而是可治理的 AI 產能。」

Salesforce、ServiceNow、Workday 都在轉向 action / credit / outcome 計價——市場要買的是可計價、可觀測、可稽核的工作產出。觀測、RBAC、預算、稽核必須是 Day 1 主產品，不是 roadmap。
